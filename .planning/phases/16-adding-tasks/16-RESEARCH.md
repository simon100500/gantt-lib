## Summary

Phase 16 adds two complementary mutations to the Gantt chart's TaskList panel: task creation and task deletion. The feature follows the "controlled component" architecture already established — the library generates an ID and emits callback data, but the parent owns the `tasks` array and mutates it.

The creation flow is a transient "ghost row" pattern: a temporary row appended to the TaskList body that is in edit mode from birth. If confirmed (Enter or blur-with-content), `onAdd` fires and the parent appends the task. If cancelled (Escape or blur-without-content), the ghost row vanishes with no side effects. This pattern is identical to the existing inline name-editing logic in `TaskListRow` — the same `useState`/`useEffect`/`nameInputRef.focus()` machinery is reused.

The deletion flow is a hover-reveal trash icon on each existing row, followed by dependency cleanup before emitting `onDelete`. The cleanup is done inside the library before firing the callback: all tasks' `dependencies` arrays are filtered to remove any reference to the deleted task ID. Both callbacks gate their buttons — if `onAdd` is absent, the "+" button does not render; if `onDelete` is absent, the trash icons do not render.

**Primary recommendation:** Implement as a "ghost row" component (`NewTaskRow`) rendered inside `TaskList` below the body, driven by `isCreating` boolean state in `TaskList`. Wire `onAdd`/`onDelete` through `GanttChart` → `TaskList` → row level.

## Architecture Patterns

### Recommended Component Structure

The cleanest split is:

```
TaskList.tsx
  ├── renders tasks.map(task => <TaskListRow ... onDelete={handleDelete} />)
  ├── renders {isCreating && <NewTaskRow onConfirm={handleConfirm} onCancel={handleCancel} />}
  └── renders {onAdd && <AddTaskButton onClick={() => setIsCreating(true)} />}

GanttChart.tsx
  ├── adds onAdd?: (task: Task) => void to GanttChartProps
  ├── adds onDelete?: (taskId: string) => void to GanttChartProps
  └── passes both to TaskList
```

### Pattern 1: Ghost Row (NewTaskRow component)

**What:** A standalone functional component that lives at the bottom of the task body, not in `tasks.map`. It holds its own `nameValue` state and an `inputRef` for auto-focus. It does NOT represent a real task — it is only a UI affordance.

**When to use:** Immediately after "+" button click. Unmounted on confirm or cancel.

```typescript
// NewTaskRow.tsx — simplified shape
interface NewTaskRowProps {
  rowHeight: number;
  onConfirm: (name: string) => void;   // caller builds the Task object
  onCancel: () => void;
}

const NewTaskRow: React.FC<NewTaskRowProps> = ({ rowHeight, onConfirm, onCancel }) => {
  const [nameValue, setNameValue] = useState('');
  const inputRef = useRef<HTMLInputElement>(null);

  useEffect(() => {
    inputRef.current?.focus();
  }, []);

  const handleKeyDown = (e: React.KeyboardEvent<HTMLInputElement>) => {
    if (e.key === 'Enter') {
      nameValue.trim() ? onConfirm(nameValue.trim()) : onCancel();
    } else if (e.key === 'Escape') {
      onCancel();
    }
  };

  const handleBlur = () => {
    nameValue.trim() ? onConfirm(nameValue.trim()) : onCancel();
  };

  return (
    <div className="gantt-tl-row gantt-tl-row-new" style={{ minHeight: `${rowHeight}px` }}>
      <div className="gantt-tl-cell gantt-tl-cell-number" />
      <div className="gantt-tl-cell gantt-tl-cell-name">
        <Input
          ref={inputRef}
          value={nameValue}
          onChange={(e) => setNameValue(e.target.value)}
          onKeyDown={handleKeyDown}
          onBlur={handleBlur}
          placeholder="Название"
          className="gantt-tl-name-input"
        />
      </div>
      {/* remaining cells empty */}
    </div>
  );
};
```

### Pattern 2: Confirm handler in TaskList — builds the Task object

```typescript
// Inside TaskList.tsx
const handleConfirmNewTask = useCallback((name: string) => {
  const today = new Date();
  const todayISO = new Date(Date.UTC(
    today.getUTCFullYear(), today.getUTCMonth(), today.getUTCDate()
  )).toISOString().split('T')[0];

  const endDate = new Date(Date.UTC(
    today.getUTCFullYear(), today.getUTCMonth(), today.getUTCDate() + 7
  )).toISOString().split('T')[0];

  const newTask: Task = {
    id: crypto.randomUUID(),
    name,
    startDate: todayISO,
    endDate,
    color: '#9ca3af',  // neutral gray (Tailwind gray-400)
  };

  onAdd?.(newTask);
  setIsCreating(false);
}, [onAdd]);
```

### Pattern 3: Dependency cleanup on delete

All cleanup happens in GanttChart before the callback fires. This keeps the library self-contained and the parent's `onDelete` simple.

```typescript
// Inside GanttChart.tsx handleDelete
const handleDelete = useCallback((taskId: string) => {
  // Remove the task and purge any dependencies referencing it
  const cleanedTasks = tasks
    .filter(t => t.id !== taskId)
    .map(t => ({
      ...t,
      dependencies: (t.dependencies ?? []).filter(d => d.taskId !== taskId),
    }));

  // Emit cleaned tasks so parent's onChange can update too
  onChange?.(cleanedTasks);
  onDelete?.(taskId);
}, [tasks, onChange, onDelete]);
```

**Note:** `onChange` should also be called with the cleaned tasks array so the parent's render stays consistent. `onDelete` carries only the id. If `onChange` is not wired, the parent must handle cleanup themselves — but calling both keeps the API consistent with existing patterns.

### Pattern 4: Hover-reveal trash on TaskListRow

The `TrashIcon` SVG is already defined in `TaskListRow.tsx` (line 30–36). Add a prop `onDelete?: (taskId: string) => void` to `TaskListRowProps`. When present, render the trash button with CSS that hides it by default and shows it on `.gantt-tl-row:hover`.

```css
/* TaskList.css additions */

.gantt-tl-row:hover .gantt-tl-row-trash {

```

### Anti-Patterns to Avoid

- **Adding `isCreating` state to GanttChart:** It belongs in `TaskList` — GanttChart should only pass `onAdd`.
- **Mutating tasks inside the library:** Library emits through callbacks; parent owns state. The `handleDelete` cleanup pattern above calls `onChange` for the cleaned array AND `onDelete` with the id — both.
- **Re-using TaskListRow for the ghost row:** The ghost row has no task object and no date cells; a separate `NewTaskRow` is cleaner than adding creation-mode to `React.memo`'d TaskListRow.
- **Calling `crypto.randomUUID()` on the server:** The `'use client'` directive is already on TaskList; this is fine. Still, build the Task in the `handleConfirmNewTask` callback which runs in browser context.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| UUID generation | Custom random ID | `crypto.randomUUID()` | Native, collision-resistant, zero deps |
| Auto-focus input | Manual DOM focus() in onClick | `useEffect(() => ref.current?.focus(), [])` | Same pattern already proven in TaskListRow (line 246-250) |
| Today's date in UTC | `new Date().toISOString()` | `Date.UTC(y, m, d)` pattern from project | Project convention: UTC-only (avoids DST bugs) |
| Trash SVG icon | New SVG | Reuse `TrashIcon` from TaskListRow.tsx (line 30-36) | Already defined, consistent styling |

## Common Pitfalls

### Pitfall 1: Ghost row causes incorrect rowHeight on the chart grid

**What goes wrong:** The ghost row in TaskList adds visual height to the TaskList panel but the chart grid does not have a corresponding row. The heights diverge — grid rows and TaskList rows desync.

**Why it happens:** The chart grid height is `tasks.length * rowHeight`. Adding a visual ghost row increases TaskList height without a matching grid row.

**How to avoid:** Position the ghost row below `gantt-tl-body` (outside the `height: totalHeight` body container). This keeps the body height matched to `tasks.length * rowHeight`. The ghost row extends beyond the body — acceptable since it is transient and below all data rows.

**Warning signs:** Rows visually misalign after clicking "+".

### Pitfall 2: Blur fires before Enter, double-confirming

**What goes wrong:** User presses Enter → `handleBlur` fires due to focus loss → `onConfirm` called twice.

**Why it happens:** `onKeyDown` (Enter) → calls `onConfirm` → parent updates `tasks` → component unmounts → `onBlur` fires during unmount sequence.

**How to avoid:** Use a `confirmedRef = useRef(false)` flag. Set it to `true` in the Enter handler before calling `onConfirm`. Check in `handleBlur`: if `confirmedRef.current`, return early.

**Warning signs:** Two tasks added on a single Enter press.

### Pitfall 3: `onDelete` called without dependency cleanup

**What goes wrong:** Deleted task's ID remains in other tasks' `dependencies` arrays. Downstream cascade logic or dependency line rendering breaks.

**How to avoid:** Always filter all tasks before calling `onDelete` (Pattern 3 above). Call `onChange` with the cleaned array first, then `onDelete`.

### Pitfall 4: TrashIcon position causes horizontal layout shift

**What goes wrong:** Adding a visible trash icon to the row pushes name/date cells left, causing layout shift on hover.

**How to avoid:** Position the trash button absolutely within the row (or in the number cell) so it does not participate in the flex layout, or use `opacity`/`visibility` (not `display`) toggling.

### Pitfall 5: `onAdd` hidden but "+" button still rendered

**What goes wrong:** If `showTaskList=true` but `onAdd` is not passed, the button should not appear. If the gate is on `showTaskList` only, the button shows when the user may not want add capability.

**How to avoid:** Gate on `onAdd` presence in `TaskListProps`. `TaskList` receives `onAdd` from `GanttChart` when it is provided in `GanttChartProps`.

## Open Questions

1. **Should `onChange` be called alongside `onDelete` for the dependency-cleaned tasks array?**
   - What we know: `onChange` handles drag/resize mutations; `onDelete` is new
   - What's unclear: If the parent wires only `onDelete` (no `onChange`), dependency orphans remain
   - Recommendation: Yes, call `onChange` with cleaned tasks before `onDelete`. Document this in JSDoc. If `onChange` is absent, still emit `onDelete` — parent must handle cleanup themselves.

2. **Gray color hex for default new task**
   - What we know: Context says "neutral gray, без акцента"; Tailwind gray-400 is `#9ca3af`
   - What's unclear: No explicit hex specified; library uses CSS variables for theming
   - Recommendation: Use `undefined` (no color) so the task renders with the default CSS variable styling, which is already neutral. Avoid hardcoding a hex unless tests show it's needed.
